# supabase setup

## migration order

1. apply `migrations/20260114_make_app_alive.sql`
2. apply `migrations/20260401_profile_images.sql`
3. run `seeds/seed_z_code_taxonomy.sql`
4. ingest partner capability output generated by:
   - `npm run data:partner-capabilities`
   - file: `sample-data/ATLASDB-excel/partner-capabilities.seed.json`

## env vars for app runtime

- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_ANON_KEY`

when these are missing, the app falls back to local seeds so development can continue without a live db.

## profile images

- Supabase Storage bucket: `profile-images`
- Metadata table: `atlas.profile_images`
- App-facing resolved URL remains on `atlas.enrollees.avatar_url`
- The trigger in `20260401_profile_images.sql` keeps `avatar_url` synced to the current primary ready image for each enrollee.
